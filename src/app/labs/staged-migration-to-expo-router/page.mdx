# Staged Migration to Expo Router without breaking releases

Migrating a live React Native app off `react-navigation` onto `expo-router` is not a weekend job. The two libraries use different mental models — declarative object graphs versus file-based routing — and the migration touches every screen, deep-link handler, and modal in the app.

I did this migration in production over several weeks, in four staged pull requests, without a single week where the team paused shipping features. This is the playbook — generic enough to apply to any team facing the same move.

## Why migrate at all

`react-navigation` is fine. `expo-router` is better *for teams that ship a lot* because it:

- **Makes URL structure explicit.** File paths *are* routes. New engineers stop needing the "how does navigation work here" tour.
- **Aligns deep-linking with the file tree.** No more separate config for URL patterns and screen definitions.
- **Enables the same app to run on web** with the same routing model.
- **Reduces boilerplate.** Screen options move to `_layout.tsx` files at the exact scope they apply.

A common trigger for the migration is cross-platform: wanting the mobile app to also serve web. `react-navigation` plus `react-navigation/web` are not the smoothest path there.

## Rule zero: don't stop shipping

Whatever staging plan you pick, the constraint is: **feature work continues in parallel**. That means:

- No feature-freeze window
- No "migration branch" that lives for weeks
- Each staged PR ships to production the same week it merges
- The app must work end-to-end after every stage

Everything downstream of that constraint is negotiable.

## The four stages

I staged the migration by *scope of the app*, from lowest-traffic areas to highest. Each stage was one PR, reviewed and merged before the next started.

### Stage 1 — a lower-traffic section first

Pick a section of the app used less often than the main flows, so any bug caught in the field has smaller blast radius. That section moves first.

- Set up an `app/<section>/_layout.tsx` with the same navigation options previously defined in the react-navigation stack.
- Port each screen file one at a time — no logic changes, just structural moves.
- Keep everything *outside* the section on react-navigation. The root navigator holds a placeholder screen that internally renders the expo-router tree.
- Both libraries coexist in the app during this stage. That is expected and OK.

The whole PR is mostly `git mv` and small type adjustments. Zero user-facing changes.

### Stage 2 — main tab tree

Once the smaller section is stable in production for a week, the main tab tree moves.

- Establish shared layout patterns (`_layout.tsx` that renders a `<Tabs>`).
- Move each tab's stack into `app/(tabs)/<name>/_layout.tsx`.
- Deep-links pointing at `/tab-name/some-detail` start working automatically because of file-based routing.

At the end of Stage 2, most user-facing routes run on expo-router. The remaining react-navigation code is the outer root navigator (auth, onboarding, splash — whichever your app has).

### Stage 3 — full app + expo-router root

The root navigator moves to `app/_layout.tsx`. This is the highest-risk stage — everything routes through it — but by now the team has used the new patterns for weeks and has good intuition.

Extra care goes into:

- **Auth guarding** via a `useAuth` hook + a top-level redirect in `_layout.tsx`.
- **Onboarding flows** that previously used imperative navigation (`navigation.replace(...)`) become `router.replace(...)`. Straight rename mostly.
- **Type-safe params.** expo-router has `useLocalSearchParams<T>()` — a small win but worth catching everywhere.

By end of Stage 3, `react-navigation` is still imported in a handful of files as a compatibility layer.

### Stage 4 — dead-code removal

Once the app has run entirely on expo-router in production for a week, the cleanup PR:

- Removes the `react-navigation` dependency.
- Deletes the old stack navigator config file.
- Deletes the compatibility layer that translated between the two libs.
- Retires any dead-code ignore-list entries that accumulated during migration.

This is the least glamorous PR of the migration and the most satisfying to merge.

## The rollback strategy

Every stage should be reversible.

- **Stages 1–2:** feature flag guards the new navigator. If a section breaks in the field, a remote-config toggle sends users back to the old react-navigation tree. Keep it on for two weeks after each stage.
- **Stage 3:** you need a full app release to roll back. Ship it during a low-visibility window so if a hotfix is needed, no user-facing launch is blocked.
- **Stage 4:** irreversible after removal. By then you should have weeks of production signal on expo-router. That is the bar.

The feature-flag approach costs a few extra branches in the code during migration but buys "if it explodes, we're fine" — worth it.

## What I'd do differently

Three things I'd change if I ran this again:

1. **Ship the compatibility layer first, on its own PR.** Bundling it into Stage 1 bloats that PR and makes review harder. A tiny "add adapter" PR before touching any screens is easier to reason about.
2. **Write the deep-link tests before Stage 1.** Retrofitting them mid-migration is possible but adds pressure. They should be the safety net from the start.
3. **Explicitly plan a "week off migration" between stages 2 and 3.** By the time you reach the root navigator, migration fatigue is real. A calm reset week catches the small bugs before they make it to production.

## Signals the migration is going well

- No spike in crash reports during the days after each stage.
- Team is using the new patterns for *new* features without being asked.
- PR sizes trending downward as boilerplate disappears.
- New engineers onboarding faster (measurable: time-to-first-shipped-PR).

None of these are obvious in week one. All four showed up by week eight. That's the payoff.

## Closing

A staged migration across four PRs, with feature-flag safety nets, produces a routing model that works on iOS, Android and web from one file tree. That is the payoff.

If you're staring at a similar migration and wondering whether to do it, the honest answer is: **it depends on how much you ship**. If your team ships two features a month, the payoff is small. If you ship ten, expo-router pays for itself in the first quarter after the migration lands.
