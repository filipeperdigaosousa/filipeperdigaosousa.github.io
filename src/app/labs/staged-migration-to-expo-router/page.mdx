# Staged Migration to Expo Router without breaking releases

Migrating a live React Native app off `react-navigation` onto `expo-router` is not a weekend job. The two libraries use different mental models — declarative object graphs versus file-based routing — and the migration touches every screen, deep-link handler, and modal in the app.

I did this migration in production over about six weeks, in four staged pull requests, without a single week where the team paused shipping features. This is the playbook.

## Why migrate at all

`react-navigation` is fine. `expo-router` is better *for teams that ship a lot* because it:

- **Makes URL structure explicit.** File paths *are* routes. New engineers stop needing the "how does navigation work here" tour.
- **Aligns deep-linking with the file tree.** No more separate config for URL patterns and screen definitions.
- **Enables the same app to run on web** with the same routing model.
- **Reduces boilerplate.** Screen options move to `_layout.tsx` files at the exact scope they apply.

For us, the trigger was cross-platform: we wanted the mobile app to also serve web. `react-navigation` and `react-navigation/web` weren't going to get us there cleanly.

## Rule zero: don't stop shipping

Whatever staging plan you pick, the constraint is: **feature work continues in parallel**. That means:

- No feature-freeze window
- No "migration branch" that lives for weeks
- Each staged PR ships to production the same week it merges
- The app must work end-to-end after every stage

Everything downstream of that constraint is negotiable.

## The four stages

I staged the migration by *scope of the app*, from lowest-traffic areas to highest. Each stage was one PR, reviewed and merged before the next started.

### Stage 1 — sessions (least-used tab)

The Sessions area of the app is used less often than home, so any bug there has smaller blast radius. Ported first.

- Set up `app/sessions/_layout.tsx` with the same navigation options previously defined in the react-navigation stack.
- Ported each session screen file one at a time — no logic changes, just structural moves.
- Kept everything *outside* Sessions on react-navigation. The root navigator held a "sessions" screen that internally rendered the expo-router tree.
- Both libraries coexisted in the app during this stage. That is expected and OK.

The whole PR was ~7 700 LOC, mostly `git mv` and small type adjustments. Zero user-facing changes.

### Stage 2 — home tabs

Once sessions was stable in production for a week, the home tab tree moved.

- Established shared layout patterns (`_layout.tsx` that renders a `<Tabs>`).
- Moved each tab's stack into `app/(tabs)/<name>/_layout.tsx`.
- Deep-links pointing at `/tab-name/some-detail` started working automatically because of file-based routing.

At the end of Stage 2, most user-facing routes ran on expo-router. The remaining react-navigation code was the outer root navigator (auth, onboarding, splash).

### Stage 3 — full app + expo-router root

The root navigator moved to `app/_layout.tsx`. This was the highest-risk stage — everything routed through it — but at this point the team had used the new patterns for weeks and had good intuition.

Extra care went into:

- **Auth guarding** via a `useAuth` hook + a top-level redirect in `_layout.tsx`.
- **Onboarding flows** that previously used imperative navigation (`navigation.replace(...)`) had to become `router.replace(...)`. Straight rename mostly.
- **Type-safe params.** expo-router has `useLocalSearchParams<T>()` — a small win but worth catching everywhere.

By end of Stage 3, `react-navigation` was still imported in a handful of files as a compatibility layer.

### Stage 4 — dead-code removal

Once the app had run entirely on expo-router in production for a week, the cleanup PR:

- Removed the `react-navigation` dependency.
- Deleted the old stack navigator config file.
- Deleted the compatibility layer that translated between the two libs.
- Retired the `knip` ignore-list entries that had accumulated during migration.

This is the least glamorous PR of the migration and the most satisfying to merge.

## The rollback strategy

Every stage was reversible.

- **Stages 1–2:** feature flag guarding the new navigator. If sessions or tabs broke in the field, a remote-config toggle would send users back to the old react-navigation tree. Kept it on for two weeks after each stage.
- **Stage 3:** we needed a full app release to roll back. We shipped it behind a marketing calendar quiet week so if we needed to hotfix, no user-visible feature was blocked.
- **Stage 4:** irreversible after removal. But we'd been running on expo-router for two months by then. Enough signal.

The feature-flag approach cost us a few extra branches in the code during migration but bought a full week of "if it explodes, we're fine" — worth it.

## What I'd do differently

Three things I'd change if I ran this again:

1. **Ship the compatibility layer first, on its own PR.** I had it inside Stage 1. It bloated the PR and made review harder. A tiny "add adapter" PR before touching any screens would have been easier.
2. **Write the deep-link tests before Stage 1.** I retrofitted them mid-migration. Should have been the safety net from the start.
3. **Explicitly plan a "week off migration" between stages 2 and 3.** By the time I got to the root navigator I was slightly overheated. A calm reset week would have caught the two small bugs that made it into Stage 3.

## Signals the migration is going well

- No spike in crash reports during the days after each stage.
- Team is using the new patterns for *new* features without being asked.
- PR sizes trending downward as boilerplate disappears.
- New engineers onboarding faster (measurable: time-to-first-shipped-PR).

None of these are obvious in week one. All four showed up by week eight. That's the payoff.

## Closing

The migration was 24k+ lines changed across four PRs, one bug rolled back via feature flag, zero incidents. The app now runs on the same routing model on iOS, Android and web — which was the point.

If you're staring at a similar migration and wondering whether to do it, the honest answer is: **it depends on how much you ship**. If your team ships two features a month, the payoff is small. If you ship ten, expo-router pays for itself in the first quarter after the migration lands.
